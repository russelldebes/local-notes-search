# local-notes-search

Semantic search and RAG (retrieval-augmented generation) over a folder of local
Markdown notes — built for an [Obsidian](https://obsidian.md) vault, but it works
on any directory of `.md` files.

Everything runs **locally**. Your notes never leave your machine: embeddings and
answers are generated by a [local Ollama](https://ollama.com) model, and the
vector index is a plain folder on disk (powered by [LanceDB](https://lancedb.com)).

## How it works

```
  your .md notes ──▶ chunk ──▶ embed (Ollama) ──▶ LanceDB vector index
                                                          │
   "your question" ──▶ embed ──▶ similarity search ───────┘
                                        │
                          ┌─────────────┴─────────────┐
                          ▼                            ▼
                 answer mode (RAG)             chunks mode
            local LLM writes an answer    raw ranked passages
              citing your notes           with scores + sources
```

On every launch the tool detects **new or changed notes** (by content hash) and
re-embeds only those — so indexing is incremental and fast after the first run.

## Prerequisites

- **macOS / Linux** (Windows via WSL)
- [**uv**](https://docs.astral.sh/uv/) — Python project + runtime manager
- [**Ollama**](https://ollama.com) — runs the local models

On a fresh Mac:

```bash
# Homebrew (if you don't have it)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

brew install uv ollama
```

## Setup

```bash
git clone <your-repo-url> local-notes-search
cd local-notes-search

# 1. Install dependencies into an isolated environment
uv sync

# 2. Point the tool at your notes
cp config.example.toml config.toml
# then edit config.toml and set [vault] path to your Obsidian vault folder

# 3. Start Ollama (in another terminal, leave it running)
ollama serve

# 4. Pull the models (first time only)
ollama pull nomic-embed-text   # embeddings
ollama pull llama3.2:3b        # answer generation (RAG)
```

If you skip step 3 or 4, the tool will tell you exactly which command to run.

## Usage

```bash
uv run notes-search
```

It indexes any changed notes, then drops you into an interactive prompt:

```
(answer) › What did I decide about the database choice?
```

### Modes & commands

| Command    | What it does                                            |
|------------|---------------------------------------------------------|
| *(type a question)* | search your notes                              |
| `/answer`  | **default** — RAG: a written answer citing your notes   |
| `/chunks`  | show the raw ranked passages with scores + sources      |
| `/clear`   | forget the current conversation (start a fresh topic)   |
| `/reindex` | rescan the vault for new/changed notes                  |
| `/stats`   | how many notes and chunks are indexed                   |
| `/help`    | list commands                                           |
| `/quit`    | exit (`Ctrl-C` / `Ctrl-D` also work)                    |

In **answer mode**, follow-up questions remember the conversation, so you can ask
"What about Russell?" and then "How tall is he?" — the tool rewrites the follow-up
into a standalone query (shown as `↳ interpreting as: …`) before searching. Use
`/clear` to start a new topic. Tune or disable this with `[chat] history_turns` in
`config.toml` (set `0` to make every question independent). Chunks mode is a plain,
stateless search.

### Just re-index, no prompt

```bash
uv run notes-search --reindex-only
uv run notes-search --vault /path/to/other/vault   # override config
```

## Configuration

All settings live in `config.toml` (copied from `config.example.toml`). You can
change the vault path, the index location, chunk size, the Ollama host, which
models to use, and how many chunks to retrieve per query. Each value can also be
overridden by a CLI flag; the vault path additionally honours the
`NOTES_VAULT_PATH` environment variable.

The vault is scanned recursively, and **hidden folders are skipped** — so
Obsidian's `.trash` (deleted notes) and `.obsidian` (config/plugins) never end up
in your index. You can also point the vault path at a parent folder containing
several vaults to search across all of them at once.

## Swapping models

There are two models, and they swap differently.

### Chat model (writes the answers in RAG mode) — easy swap

1. Pull the model you want (one-time download):
   ```bash
   ollama pull qwen2.5:7b
   ```
2. Set it in `config.toml`:
   ```toml
   [ollama]
   chat_model = "qwen2.5:7b"
   ```
3. Restart the tool. That's it — **no re-indexing needed.**

> You can skip step 1: if you set a model you haven't pulled, the tool detects it
> on launch and prints the exact `ollama pull …` command to run.

Larger models generally give better, more robust answers at the cost of more disk,
RAM, and latency. Good local options:

| Model            | Size    | Notes                                             |
|------------------|---------|---------------------------------------------------|
| `llama3.2:3b`    | ~2 GB   | Default. Fast and light; fine for simple recall.  |
| `llama3.1:8b`    | ~4.7 GB | Noticeably stronger synthesis.                    |
| `qwen2.5:7b`     | ~4.7 GB | Strong instruction-following; great all-rounder.  |

Browse more at [ollama.com/library](https://ollama.com/library).

### Embedding model (turns notes into vectors) — triggers a re-index

Embeddings from different models live in different, incompatible vector spaces, so
an existing index can't be searched with a new model's vectors. The tool guards
against this automatically: it records which embedding model built the index, and
if it sees `embed_model` change, it **rebuilds the whole index from scratch** on
the next run. So swapping is still just:

```bash
# 1. Pull the new embedding model
ollama pull mxbai-embed-large

# 2. Set it in config.toml
#    [ollama]
#    embed_model = "mxbai-embed-large"

# 3. Run the tool — it detects the change and re-embeds every note
uv run notes-search
```

You don't need to delete anything by hand — the mismatch is caught for you. (If you
ever want a clean slate anyway, `rm -rf .notes_index` and re-run is always safe.)

## On resource usage

There's no database server and no containers. The vector index is just files in
`.notes_index/`. Ollama runs as a lightweight background service and
automatically unloads model weights from memory after a few minutes of inactivity
(its `keep_alive` default), so it isn't holding RAM/VRAM when you're not searching.
